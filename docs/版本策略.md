# 版本策略

本文档记录 `sb6657.cn` 第三版网站的版本号规则、历史版本溯源和后续升级口径。自 `V3.13.11.20260721` 起启用新规则；此前 `YY.MM.DD` 格式的更新日志作为旧版本记录保留，不做批量改写。

## 版本号格式

面向用户和更新日志的完整版本号使用：

```text
Vmajor.minor.patch.yyyymmdd
```

- `major`：网站大版本。只有整体产品代际或不兼容的重大迭代才增加，由项目负责人决定。
- `minor`：大块功能版本。新增独立业务模块、核心体系，或让已有模块发生能力级跃迁时增加。
- `patch`：修订版本。Bug 修复、UI/交互优化、重构、工程调整和未达到 minor 门槛的功能增强都增加 patch。
- `yyyymmdd`：实际发布当天的本地日期，固定为 8 位，例如 `20260721`。

项目不使用 `const` 构建号和 `_class` 阶段后缀。`major`、`minor`、`patch` 是互相独立的整数，不是小数位。

## 代码中的存储形式

完整展示版本与 npm 元数据表达的是同一个版本，但受 npm SemVer 格式约束，序列化方式不同：

| 位置 | 格式 | 当前示例 |
| --- | --- | --- |
| 页面右下角、`sbVersion`、源码更新日志标题 | `Vmajor.minor.patch.yyyymmdd` | `V3.13.11.20260721` |
| `package.json`、`package-lock.json` | `major.minor.patch+yyyymmdd` | `3.13.11+20260721` |

`+yyyymmdd` 是标准 SemVer 的构建元数据写法。不要把四段点分版本或前缀 `V` 直接写进 `package.json.version`。

C 端展示时，右下角直接显示完整版本，不再额外添加“版本：”前缀。更新日志时间线顶部显示发布日期；下方标题对旧版本保留 `版本【YY.MM.DD】`，对新版本显示不重复日期的 `版本【Vmajor.minor.patch】`。

## 升级规则

每个准备发布的代码改动批次按以下优先级判断：

1. 产品代际或不兼容重大迭代：`major + 1`，`minor` 和 `patch` 归零。
2. 新增大块功能：`minor + 1`，`patch` 归零。
3. 其他代码改动：`patch + 1`。
4. 日期始终换成实际发布当天的 `yyyymmdd`。

一次任务或一次发布批次包含多项改动时只升级一次，按其中最高级别决定。一天内发布多个独立批次时，每个批次继续递增 patch，日期可以相同。纯内部文档改动默认不升级版本；用户可见发布文档、版本规则调整或用户明确要求时除外。

### minor 判定口径

通常计入 minor：

- 新增具有独立入口、独立数据或完整交互闭环的业务模块。
- 新增跨页面的核心平台能力，例如账户体系、实时连接体系。
- 重做核心信息架构或数据模型，并显著改变主要使用方式。
- 已有模块新增足以改变能力层级的大型子系统，例如 AI 从单轮生成升级为带记忆的会话体系。

通常只计入 patch：

- Bug 修复、样式调整、响应式适配、性能优化和代码重构。
- 排序、筛选、导出、入口、提示、归档、展示字段等既有模块增强。
- 工程化、依赖、CI、文档和开发流程调整。
- 季节性视觉效果、彩蛋、外部链接和未形成完整业务模块的小功能。

边界不清时先结合 `docs/更新日志.md` 和 Git 历史判断，再由用户或项目负责人决定；不要仅根据 commit message 中是否出现 `feat` 自动增加 minor。

## 修改代码时的同步清单

后续 AI 或开发者修改代码时必须：

1. 读取当前本地日期和本文件中的当前版本。
2. 按上述口径决定 major、minor 或 patch。
3. 在 `docs/更新日志.md` 新增 `## 版本【Vmajor.minor.patch.yyyymmdd】`，并写简短改动项。
4. 将 `package.json` 和 `package-lock.json` 根包版本同步为 `major.minor.patch+yyyymmdd`。
5. 将 `src/apis/httpInstance.ts` 的 `sbVersion` 同步为完整展示版本 `Vmajor.minor.patch.yyyymmdd`。
6. 运行项目要求的 build/lint，并检查四处版本表达的 major、minor、patch 和日期完全一致。
